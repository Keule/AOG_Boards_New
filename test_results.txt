3. app_core lässt keinen produktiven Zustand ohne vollständige RTCM-Ausgabe zu
4. rtcm_router_add_output()-Rückgabewerte werden geprüft
5. Router bleibt generisch (kein Boardprofil/HAL/UART/NTRIP im Router)
6. Drop-/Backpressure-Policy generisch getestet
7. Boardprofile produktiv abgesichert (Compile-Smoke)
8. Wiring-Logik als testbare pure Funktion extrahiert
9. Doku angepasst
10. Known Issue NTRIP dokumentiert

ARCHITECTURE DECISIONS
======================

1. RTCM-Zielpfad:      Generisch — Alle aktiven GNSS-Empfänger
2. RTCM-Format:        Variante A — Transparenter Byte-Stream (passthrough)
3. Drop-Policy:        Variante A — Drop erlaubt bei Output-Overflow
4. Router-Outputs:     Gekoppelt an board_profile_has_uart_tx()
5. Wiring:             nav_rtcm_wire_outputs() pure Funktion
6. Keine starre 2:     Generische Iteration über GNSS-Port-Tabelle

PIN ASSIGNMENTS (Board Profile)
==============================
ESP32:    UART_NUM_1 TX=GPIO14 RX=GPIO16  → UM980 #1
          UART_NUM_2 TX=GPIO15 RX=GPIO17  → UM980 #2
ESP32-S3: UART_NUM_1 TX=GPIO6  RX=GPIO4   → UM980 #1
          UART_NUM_2 TX=GPIO7  RX=GPIO5   → UM980 #2

KEY NEW COMPONENTS
==================
- nav_rtcm_wiring.h/.c: Pure function for generic RTCM output wiring
  - nav_rtcm_wire_outputs(targets, count, router) → wiring_result_t
  - Checks: has_uart, has_uart_tx, tx_buffer != NULL, add_output return
  - Returns productive=true only if all active receivers are wired

- board_profile: GNSS Port Iteration API
  - board_profile_get_gnss_port_count() → int (currently 2)
  - board_profile_get_gnss_port(index) → board_uart_port_t
  - Static table: s_gnss_ports[] — extensible with one line per receiver
