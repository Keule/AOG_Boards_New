# PGN 214 — Consolidated GPS + Heading Output (AOG v5)

> **PGN:** 214 (0xD6) · **Direction:** GPS Module → AgOpenGPS / AgIO · **Rate:** Up to 20 Hz

---

## 1. Overview

PGN 214 is the primary output PGN from the GPS navigation module to AgOpenGPS (via AgIO). It consolidates position, heading, speed, altitude, and GNSS quality metrics into a single binary frame, eliminating the need for multiple sentence types.

The module produces this frame at up to 20 Hz when both GNSS and heading sources are valid and fresh. When data is invalid or stale, sentinel values are substituted so that the consumer can unambiguously detect the failure mode.

---

## 2. Frame Format (v5)

Every PGN uses the same framing envelope:

```
Byte Position  Field        Size  Value / Notes
─────────────────────────────────────────────────────────────
0              PREAMBLE_0   1     0x80
1              PREAMBLE_1   1     0x81
2              SRC          1     Source address (0x05 = GPS module)
3              PGN_LO       1     PGN low byte  (0xD6 for PGN 214)
4              PGN_HI       1     PGN high byte (0x00 for PGN 214)
5              LEN          1     Payload length in bytes (51)
6..56          PAYLOAD      51    See §3
57             CRC          1     Sum of bytes [2..56] mod 256
```

### Key fields

| Field | Description |
|-------|-------------|
| **Preamble** | Fixed `0x80 0x81`. Used by the receiver for frame sync and byte-stuffing detection. |
| **SRC** | Source address. `0x05` for the GPS module. Other modules: `0x06` IMU, `0x07` Steering. |
| **PGN** | 16-bit little-endian PGN identifier. For PGN 214 this is `0xD6 0x00`. |
| **LEN** | Payload byte count. Always `51` (0x33) for PGN 214 v5. |
| **CRC** | 8-bit checksum covering bytes from SRC (index 2) through the last payload byte (index 56), inclusive. Computed as `sum(bytes[2..56]) & 0xFF`. |

**Total frame size:** 59 bytes.

---

## 3. Payload Structure (51 bytes)

All multi-byte fields are **little-endian**.

| Offset | Size | Type | Field | Unit | Description |
|--------|------|------|-------|------|-------------|
| 0 | 8 | `f64` | `longitude` | degrees | Decimal degrees, East positive |
| 8 | 8 | `f64` | `latitude` | degrees | Decimal degrees, North positive |
| 16 | 4 | `f32` | `heading_dual` | degrees | Dual-antenna heading, 0–360 |
| 20 | 4 | `f32` | `heading_true` | degrees | Course over ground, 0–360 |
| 24 | 4 | `f32` | `speed` | km/h | Ground speed |
| 28 | 4 | `f32` | `roll` | degrees | Vehicle roll |
| 32 | 4 | `f32` | `altitude` | meters | Altitude above geoid |
| 36 | 2 | `u16` | `satellites` | count | Satellites used in solution |
| 38 | 1 | `u8` | `fix_quality` | enum | Fix quality code (see §5) |
| 39 | 2 | `u16` | `hdop` | ×0.01 | Horizontal dilution of precision (e.g., 1.5 → 150) |
| 41 | 2 | `u16` | `age` | ×0.01 s | Differential correction age |
| 43 | 2 | `u16` | `imu_heading` | ×0.1 deg | IMU-derived heading |
| 45 | 2 | `i16` | `imu_roll` | ×0.1 deg | IMU roll |
| 47 | 2 | `i16` | `imu_pitch` | deg | IMU pitch (no scaling) |
| 49 | 2 | `i16` | `imu_yaw_rate` | deg/s | IMU yaw rate (no scaling) |

### Memory layout (hex offsets)

```
00  07 │ f64  longitude
08  0F │ f64  latitude
10  13 │ f32  heading_dual
14  17 │ f32  heading_true
18  1B │ f32  speed
1C  1F │ f32  roll
20  23 │ f32  altitude
24  25 │ u16  satellites
26      │ u8   fix_quality
27  28 │ u16  hdop
29  2A │ u16  age
2B  2C │ u16  imu_heading
2D  2E │ i16  imu_roll
2F  30 │ i16  imu_pitch
31  32 │ i16  imu_yaw_rate
```

## 4. Sentinel Values

Sentinel values signal that a field contains no valid measurement. The consumer must check for these values before using the data.

| Type | Sentinel | Notes |
|------|----------|-------|
| `f64` (lat, lon) | `1.7976931348623157e+308` (`double.MaxValue`) | IEEE 754 all-bits-set exponent + mantissa |
| `f32` (heading, speed, roll, altitude) | `3.4028235e+38` (`float.MaxValue`) | IEEE 754 `0x7F7F_FFFF` |
| `u16` | `0xFFFF` (65535) | e.g., satellites, hdop, age, imu_heading |
| `i16` | `0x7FFF` (32767) | e.g., imu_roll, imu_pitch, imu_yaw_rate |

> **Rule:** A sentinel in *any* field does **not** invalidate the remaining fields. Each field is independently valid or sentinel.

## 5. Fix Quality Mapping

The `fix_quality` byte is translated from the NMEA GGA fix-quality indicator:

| NMEA GGA | AOG PGN 214 | Enum Label | Description |
|----------|-------------|------------|-------------|
| 0 | 0 | `NONE` | No fix |
| 1 | 1 | `GPS` | Autonomous GPS |
| 2 | 2 | `DGPS` | Differential GPS (SBAS, beacon, etc.) |
| 4 | 4 | `RTK_FIX` | RTK Fixed (integer) |
| 5 | 5 | `RTK_FLOAT` | RTK Float |
| other | 0 | `NONE` | Unknown / unsupported — treated as no fix |

> Values 3 and 6–255 are not defined in the AOG protocol and map to `0` (NONE).

---

## 6. Output Gating Rules

The module applies gating logic before emitting a PGN 214 frame:

1. **GNSS must be valid AND fresh.**
   - *Invalid* (no fix, `fix_quality == 0`) → full sentinel frame with `fix_quality = 0`.
   - *Stale* (last update older than the freshness threshold, default 2000 ms) → full sentinel frame with `fix_quality = 0`.
2. **Heading is gated independently.**
   - If heading is *invalid* or *stale* (> 3000 ms without update), `heading_dual` and `heading_true` are set to `float.MaxValue`, regardless of GNSS state.
3. **No partial reuse.** A stale heading value is never retransmitted; once stale, sentinel is emitted until a fresh heading arrives.

### Decision matrix

| GNSS | Heading | Position | Heading fields | fix_quality |
|------|---------|----------|----------------|-------------|
| Invalid | — | sentinel | sentinel | 0 |
| Stale | — | sentinel | sentinel | 0 |
| Valid | Invalid | real | sentinel | from GGA |
| Valid | Stale | real | sentinel | from GGA |
| Valid | Valid | real | real | from GGA |

## 7. CRC Calculation

The CRC is an 8-bit additive checksum covering all bytes from the **SRC** byte (index 2) through the **last payload byte** (index 56). It does **not** include the preamble (bytes 0–1) or the CRC byte itself.

```
CRC = (SRC + PGN_LO + PGN_HI + LEN + PAYLOAD[0] + ... + PAYLOAD[50]) & 0xFF
```

### Worked example

Consider a minimal (truncated for illustration) frame where only the header bytes are shown:

```
0x80  0x81  0x05  0xD6  0x00  0x33  [51 bytes of payload]  CRC
```

Compute CRC over bytes at indices 2..56:

```
CRC = (0x05 + 0xD6 + 0x00 + 0x33 + payload_byte_0 + ... + payload_byte_50) & 0xFF
    = (0x05 + 0xD6 + 0x00 + 0x33 + Σ(payload)) & 0xFF
```

If the payload were all zeros:

```
CRC = (0x05 + 0xD6 + 0x00 + 0x33 + 0) & 0xFF
    = 0x10E & 0xFF
    = 0x0E
```

The CRC byte appended at index 57 would be `0x0E`.
